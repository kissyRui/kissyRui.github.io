不是你努力决心没下，而是你惰性太大[摘录]

发表于 2016-06-07   |   分类于 百家   |  

俗话说“少壮不努力，老大徒伤悲”，其实努力不应该仅在“少壮”时期，人的一生都应该不停努力，努力工作、努力生活、努力玩儿。人的整个成长过程就是一个努力的过程，只是程度不同而已。

努力才能进化

从进化来讲，努力的过程就是从非条件反射到条件反射的过程。非条件反射是天生的，不用学习就会的如人生下来就会吃奶，给婴儿一个手指放到嘴边，也会做出同样的吸吮动作。条件反射是后天形成的，是经过学习或努力适应环境得来的，如学会游泳和学会骑自行车要经过多次学习和练习才能得心应手。非条件反射是有限的而条件反射是无限的，条件反射扩展了机体对外界复杂环境的适应范围，使机体具有更大的预见性、灵活性和适应性。

当你经过努力对某个知识或技能形成条件反射的时候，意味着你对此知识或技能已轻车熟路，进入了自己的舒适区，进入了自动模式。这个区域没有挑战，没有挑战就没有进步，你就get不到新的知识和技能，慢慢的就out了。社会和环境在不断变化，你如果不努力适应，也许会被社会所淘汰。努力分成主动努力和被动努力。一个人如果没有前瞻性和预见性，当环境发生变化时再开始努力就要付出加倍的辛苦。因此与其“亡羊补牢”，不如早做准备。

努力程度=驱动力-惰性

人的努力程度与驱动力成正比，与惰性成反比。所有从非条件反射到条件反射都需要努力，这就是人的痛处，英文叫‘no pain, no gain’。因此很多人自动走到自己的舒适区，不能知难而进，放弃努力也就远离成功。同时人都有惰性，一方面要和自己的惰性作斗争，另一方面更重要的是提高自己的驱动力。驱动力来自于自己对梦想和目标的追求，来自于不断努力的成果和一个个的成就带来的激励。梦想可以驱动你向前走，目标驱动着你在前行的路上不会走错路。有梦想才有动力，有目标才能前行。

努力不要做“差不多先生”

同样是努力但程度不同。有的人达到60分的结果，有的人达到90分的结果。想要追求100分，则要精益求精，而不能“差不多就行了”。这也是一般人和专家的区别。一般人把某种技能精进当作工作的附加品，顺便掌握一些经验和技能，能完成任务则万事大吉。但顶尖专家则会不断追求卓越，超越他人，把一件事做到极致。现在提倡的工匠精神就是这个意思。他要的不光是技能，还要学习独到的远见和眼光，总是力争能比别人看的更远一步，做得更努力一点。人的能力取决于自己的能力库。库里的能力项目越多，能力越强。能力有多方面的表现如硬实力和软实力。硬实力如技能，软实力如洞察力和利用资源的能力等。人越努力，建立的条件反射越多，能力库的项目越多，适应力和创造力等越强，综合起来就是能力越强。

稻盛和夫曾说过，凡是功成名就的人毫无例外地都是不懈努力，历尽艰辛，埋头苦干自己的事业，才取得巨大成功。通过艰苦卓绝的努力，在成就伟大功绩的同时，他们也造就了自己完美的人格。只有不断挑战自己，勇于创新，解决新问题，主动学习新技术，不随波逐流，才能到达美好的彼岸。

React Native支持的特性

发表于 2016-05-26   |  

本调研报告基于Android，我为什么这么做呢？基于React Native最先支持iOS平台，所以相对比较稳定成熟；而Android想对来说支持晚，基于Android平台来调研更能看出React Native所支持的特性。

Web的用户体验仍无法超越Native：

  • Native的原生控件有更好的体验
  • Native有更好的手势识别
  • Native有更合适的线程模型，尽管Web Worker可以解决一部分问题，但如图像解码、文本渲染仍无法多线程渲染，这影响了Web的流畅性

多平台支持

目前对iOS支持很完善，Android也基本上还可以满足需求

开发成本低，效率高

相对而言，开发成本更低，代码量更少

性能优于Web，体验流畅

也是为了解决webview的性能问题而生，实际使用体验上还是很流畅的

组件化编程体验

react天生就是拥有组件化思想的框架

Redux框架

可定制化UI组件

支持的UI组件提供了很大程度的扩展性，如Modal

地理定位

navigator.geoloaction的支持
地理坐标经纬度支持，可以做定位需求

同时，在Android AndroidManifest.xml中配置支持地理地位

相册接口

提供手机相册的读取和保存，当然可以利用摄像头

剪切板

传统webview内不支持的复制功能RN支持了

异步本地存储

类似于localstorage，可以实现购物车等类似功能

获取应用状态

可以根据应用状态做一些处理

webview的扩展支持(Android变通)

对于一些活动类的页面，RN也支持webview组件

消息推送机制

可以利用原生的调用实现消息推送

可以设置自定义字体

可以设置自定义字体，建议引入ttf格式

键盘支持

对键盘的支持，可以接近原生

动画支持

setTimeout，setInterval，Animated，css3，requestAnimationFrame，调用原生动画效果等等方式可以用

原生与JS通信机制

js和原生通信很重要

原生视图和模块的扩展

这个可以发挥原生优秀UI组件或模块的优势

网络状态的检测

能检测网络的状况

平滑的页面切换

navigator页面跳转很平滑

热更新特性

动态更新线上业务代码，避免过多苹果APP Store审核

缓存/打包

做前端，已五年

发表于 2016-05-14   |  

Think Different, Do Great

我的大学

大学里其实对前端有一些想法，学了Ps，jQuery，Css等。
主要学了不少关于Flash As3的东西，以为会很火，结果乔帮主说，Flash终将死去，然后我就忍痛放弃了。

2011年5月20日［喜团网］

对我来说，这是一个不同寻常的日子。我入职了一家叫喜团网的团购网站，当时正是千团大战时，当时前端开发岗位也略热。
遇到了淘宝前端大牛玉伯开发的kissy框架，该框架第一次让我认识到前端也可以有封装良好的UI和业务组件。
当时遇到了两个前端前辈黄兄和虫哥，并受益匪浅。
三个月后，团购网站激烈的竞争导致喜团资金链出问题，大家频频离职：黄兄去了淘宝网，虫哥去了支付宝。我经过黄兄介绍，让我去点评网试试。

2011年11月1日［大众点评］

这天入职大众点评网，开始了近两年的前端光阴。
当时的点评产品分为两个线：主站和点评团购，APP还是九宫格。
这是一个仅有9个前端攻城狮的O2O公司，团购业务只有2个前端来负责跟进迭代，由于团购的激烈竞争，点评也加大了团购产品的投入，所以我被分配到了团购前端小组。
当时团购用的框架是Mootools，一个产品最初的前端框架都是很仓促的，一般是根据执行者对框架的熟悉程度定的。Mootools是一个不错的库，代码质量很不错，写起来很有原生的感觉。
点评团主站页面很多了，随着多人开发模式的业务特点，点评前端架构师张颖正着力开发一个AMD标准的库NeuronJs
2012年上半年，团购站需要从.Net迁移到java，我和另外一个伙伴就利用这个契机完成了站点从Mootools到Neuron的代码迁移。
下半年，随着手机应用的增多，超哥启动了一个新的项目（点评触屏版）单独让我负责。我记得当时我们用的还是Wap，基本上没啥JS代码。
但是对H5还不是很了解，硬着头皮上，当时选择了小巧的Zepto移动框架，之前我写HTML不是特别多，但这个项目结构样式都由我来写，所以相对来说，我接触H5算是比较早的，遇到的坑也是蛮多的，市面上各种低端机，什么魅族M8，索爱LT系列，苹果4等等。
后来基本上线后，又开始了触屏版团购版和用户登录注册模块，但相对顺手了一些，不过跨团队工作也蛮辛苦的。
2013年春站点又经历了三四次功能和UI迭代开发，当时自己负责的业务模块有：

  • 团购个人中心
  • 团购验券商户平台
  • 团购触屏版
  • 主站触屏版
  • 点评登录注册模块触屏版
  • 后来又有新的项目立项，自己又承担了点评优惠券及其新的优惠券管理平台的开发。

2013年年初和年中，面对每天这么多同步的开发，而且跨部门，跨楼层的沟通，对自己是一个极大的考验，我试着把自己每一天的工作都列出来，然后一项项完成打勾，尽管如此，有些产品线还是有些delay了时间。
当时我定在十月份结婚，七八月份时候跟爱人沟通少了很多，面对爱人的抱怨，我开始反思自己的问题；虽然工作很重要，自己也愿意给自己一些压力，可是工作不是一切，生活和感情也是至关重要的，我选择离开点评，我选择一份专门做移动H5的工作。回望过去，我在点评这段时光恰恰是成长的一段时光。

2013年8月6日［盛大游戏］

我对盛大游戏充满很深的感情，盛大见证了我的婚姻，也见证了我家童童的出生。
2013年8月6日，我进入盛大，在老曹的手下做H5前端，刚来的时候就遇到了WE框架，我不能不说这是个很不错的流程类的框架，依赖jQuery和requireJs两个类库，对于流程的理解和组件的处理方式至今让我觉得还是很先进。这个框架也可以移植在很多应用环境中，但是对于UI的处理分离度不够，对于移动端改造有些问题。
话说，当时9月份就遇到了一个项目，G家商城H5版本，这个项目也许是我在盛大的第一个H5作品，而且是被派到技保中心来开发，作为一个小型商城，也没有采用过于复杂的框架，为了考虑以后的扩展性及其组件上的积累，使用了Seajs+zepto
后来部门搬到了3楼南，在2014年的一年里，我记不清到底做了多少个项目了：

  • 跟陈洋一起折腾FF14各种活动，才知道不折腾下就不知道自己能力在哪里
  • 跟佩佩、亚平、龙哥一起做G家APP内部运营活动，才知道移动H5开发合作的不易与做活动所需要的耐心
  • 跟传君一起做锁链战记等手游活动，才知道有时候加班也是没办法的办法
  • 跟夔叔一起做G家通用导航条，才知道让别人用起来爽，自己多写点代码没啥
  • 跟神父一起做红金宝活动，才知道运营其实也蛮考验人的，需要很严谨，很认真

等等…
一幕幕恰如昨日发生，记忆犹新。
这一年显得忙碌又充实，在参考别人的同时，更新自己的技术，玩过AngularJS，响应式，Css3动画，Less，Sass，Grunt，Gulp一系列新的东西，受益匪浅。

2015年1月9号［房金所］

2015年还没过年，就在一个不合适的时间点去了一家P2P公司－房金所，在这家公司好像一切都要从头开始搞：

  • 先是后台管理采用了requirejs的改进，大大提升了使用体验
  • H5移动触屏版（将来的App版本），面对自己熟悉的框架，我选择了没有接触过的Reactjs+Flux，还有新认识的朋友Webpack；这个项目做下来感觉还是很花精力的，不过对于Store的处理，个人还是觉得Flux做得不到位，后来又改成了Redux框架
  • 主站要全面改版，当时遇到了百度团队的FIS，很棒的一个东西，它采用了commonjs的方式来写浏览器代码，很多优秀的特性；对于这次架构和各个模块的处理，感觉还是不错的
  • 搞了几次特别的理财和抽奖活动

2015年8月9日［盛大游戏］

我又回到了盛大游戏，内心倍感温暖！
完成了一系列站点的响应式改造，由于现在比较流行采用Commonjs的规范，所以整体都采用这个，就像写Nodejs一样写浏览器端代码。

  • SDO官网
  • 通行证的PC/响应式同步改造
  • 实现了We的初步响应式
  • 公共导航
  • 盛大游戏WEB版登录、注册改版
  • 参与了盛大游戏官网
  • 游戏详情页如血族
  • 手游的头部Pc和移动导航条如血族
  • 开发了微信账号管理和角色管理项目，升级了we为Mobile We（简单介绍下Mobile We的角色管理）
    正在进行时…

最近在研究React Native，希望能够用在公司产品上
做前端这么多年，说实话很喜欢，这里充满变化，充满惊喜，充满挑战
现在大多数公司又比较重视用户体验
自己能做的就是不断学习，跟得上技术更新的潮流，最最重要的是要学有所用才能真才实学。
如何成为优秀的前端呢？其实我有个比较好的前端朋友-寸志，可以了解下他。
